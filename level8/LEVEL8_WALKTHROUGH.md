# Level 8 Walkthrough

This is the single interactive guide for Level 8. It covers:

- starting the Docker Compose demo step by step
- watching the audit logs
- running a Python MCP-style agent on the host with `uv`
- registering that Python agent with an existing registry
- creating a new dynamic registry
- registering an agent with the new registry
- finding the registry through NANDA and calling the tool with `curl`

There are two roles in this walkthrough:

- **Producer side**: you run or create agents/registries and register them so they become discoverable.
- **Client side**: you act as a user of NANDA with `curl`, searching the index, resolving signed registry address credentials, fetching signed facts, and calling a discovered tool.

Registration does not happen directly from an agent to NANDA in this demo. Agents register with a registry through `POST /agents/register`; the registry signs AgentFacts-like catalog data and refreshes its NANDA `/join` record. NANDA stays a lean discovery index with compact registry and agent summaries.

For only the client-side discovery flow, use `CURL_CLIENT_GUIDE.md`. This walkthrough includes both producer-side setup and client-side checks.

Run commands from this directory unless noted:

```sh
cd level8
```

## 1. Clean And Build

```sh
# Stop old Level 8 containers and remove old compose network attachments.
docker compose down --remove-orphans

# Optional but recommended for a fresh walkthrough.
# These directories are generated by the demo and ignored by git.
rm -rf artifacts logs bin
mkdir -p artifacts logs bin

# Build the shared Ubuntu-based image used by every modeled component.
docker compose build

# Compile every Go component into ./bin inside the shared Ubuntu image.
docker compose up go-build

# Generate local artifacts:
# self-signed certs, issuer keys, trust bundle, W3C-style VCs,
# registry addr credentials, signed catalogs, and status-list data.
docker compose up artifact-init
```

## 2. Start The Base Quilt

```sh
# Start two federated NANDA indexes.
docker compose up -d nanda-index-a nanda-index-b

# Start security and update services.
docker compose up -d \
  credential-rotator \
  issuer-key-rotator \
  revocation-authority \
  crdt-update-bus \
  private-facts-gateway

# Start the built-in enterprise registries.
# Enterprise A joins NANDA A; Enterprise B joins NANDA B.
docker compose up -d enterprise-a-registry enterprise-b-registry

# Start the built-in MCP agents advertised by the signed catalogs.
docker compose up -d \
  enterprise-a-reverse \
  enterprise-a-uppercase \
  enterprise-b-truncate \
  enterprise-b-count

# Start the dynamic client/verifier.
docker compose up -d consumer

# Optional: start the compact audit swimlane.
# It reads ./logs/*.jsonl and prints a summarized view.
docker compose up -d swimlane
```

Check the stack:

```sh
# Shows running containers and host port mappings.
docker compose ps

# Optional: follow the swimlane summary in another terminal.
# Press Ctrl-C to return; the stack stays running.
docker compose logs -f swimlane

# NANDA A knows NANDA B as a federated peer.
curl --resolve nanda-a.local:18080:127.0.0.1 http://nanda-a.local:18080/peers

# NANDA A can search joined registries.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  'http://nanda-a.local:18080/search?registrationType=enterprise-mcp-registry'
```

## 3. Watch Real Activity

Most demo activity is written to JSONL audit files under `./logs`. `docker compose logs` mostly shows process stdout/stderr and startup errors.


## 4. Create A Host Python Agent

The Python agent is not discoverable just because it is listening on a port. It becomes discoverable only after a registry signs facts that advertise it and refreshes its NANDA join.

Install or verify `uv`:

```sh
# Proves uv is already available.
uv --version

# If uv is missing on macOS, install it with Homebrew.
brew install uv

# Install a Python runtime managed by uv if needed.
uv python install 3.12
```

Create a tiny MCP-style Python agent:

```sh
cat > /tmp/level8_python_agent.py <<'PY'
#!/usr/bin/env python3
# /// script
# requires-python = ">=3.12"
# dependencies = []
# ///
from http.server import BaseHTTPRequestHandler, ThreadingHTTPServer
import json
import os

AGENT_ID = os.environ.get("AGENT_ID", "host-python-tools")
TOOLS = [tool.strip() for tool in os.environ.get("TOOLS", "echo,words").split(",") if tool.strip()]


class Handler(BaseHTTPRequestHandler):
    def _json(self, status, payload):
        body = json.dumps(payload).encode("utf-8")
        self.send_response(status)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

    def do_GET(self):
        if self.path == "/healthz":
            self._json(200, {"ok": True, "service": AGENT_ID})
            return
        if self.path == "/mcp/tools/list":
            self._json(200, {
                "agent": AGENT_ID,
                "tools": [{"name": tool, "description": f"Python demo tool: {tool}"} for tool in TOOLS],
            })
            return
        self._json(404, {"error": "not found"})

    def do_POST(self):
        if self.path != "/mcp/tools/call":
            self._json(404, {"error": "not found"})
            return
        length = int(self.headers.get("Content-Length", "0"))
        request = json.loads(self.rfile.read(length) or b"{}")
        tool = request.get("tool")
        text = request.get("input", "")
        if tool not in TOOLS:
            self._json(400, {
                "agent": AGENT_ID,
                "requested_tool": tool,
                "allowed_tools": TOOLS,
                "error": "skill mismatch",
            })
            return
        if tool == "words":
            result = str(len(text.split()))
        elif tool == "title":
            result = text.title()
        elif tool == "length":
            result = str(len(text))
        else:
            result = text
        self._json(200, {
            "agent": AGENT_ID,
            "tool": tool,
            "input": text,
            "result": result,
        })

    def log_message(self, _format, *_args):
        return


if __name__ == "__main__":
    port = int(os.environ.get("PORT", "8080"))
    ThreadingHTTPServer(("", port), Handler).serve_forever()
PY
```

Start it in a separate host terminal:

```sh
AGENT_ID=host-python-tools TOOLS=echo,words PORT=19090 uv run --python 3.12 /tmp/level8_python_agent.py
```

Smoke test it:

```sh
# Proves the Python process is alive.
curl http://127.0.0.1:19090/healthz

# Proves it exposes the MCP-style tool list.
curl http://127.0.0.1:19090/mcp/tools/list

# Proves the echo tool works.
curl -s -X POST http://127.0.0.1:19090/mcp/tools/call \
  -H 'Content-Type: application/json' \
  -d '{"tool":"echo","input":"hello from host python"}'

# Proves skill mismatch rejection works.
curl -s -X POST http://127.0.0.1:19090/mcp/tools/call \
  -H 'Content-Type: application/json' \
  -d '{"tool":"not-registered","input":"should fail"}'
```

## 5. Register The Python Agent With Enterprise A

This registers the host Python agent into Enterprise A's signed AgentFacts-like catalog. The endpoint uses `host.docker.internal` so Docker containers can call back to the host process.

On Docker Desktop for macOS, `host.docker.internal` is built in. The Level 8 `consumer` service also maps it through Compose so the Docker client can call host-run agents.

```sh
# Registers the Python agent with Enterprise A.
curl --resolve enterprise-a.registry.nanda.local:18081:127.0.0.1 \
  -X POST http://enterprise-a.registry.nanda.local:18081/agents/register \
  -H 'Content-Type: application/json' \
  -d '{
    "id": "host-python-tools",
    "name": "host.python.tools.mcp.local",
    "endpoint": "http://host.docker.internal:19090",
    "tools": ["echo", "words"]
  }'

# Shows Enterprise A's static plus dynamically registered agents.
curl --resolve enterprise-a.registry.nanda.local:18081:127.0.0.1 \
  http://enterprise-a.registry.nanda.local:18081/agents
```

Wait two seconds for Enterprise A to refresh its NANDA join.

```sh
# Finds the registry that advertises the new Python echo tool.
# The response includes agentMatches with compact endpoint metadata.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  'http://nanda-a.local:18080/search?tool=echo'

# Finds the same Python agent by agent id.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  'http://nanda-a.local:18080/search?agent=host-python-tools'

# Resolves the signed registry address credential from NANDA.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  http://nanda-a.local:18080/resolve/enterprise-a.registry.nanda.local

# Fetches the signed catalog that now includes host-python-tools.
curl --resolve enterprise-a.registry.nanda.local:18081:127.0.0.1 \
  http://enterprise-a.registry.nanda.local:18081/catalog

# Calls the registered Python echo tool from the host.
curl --resolve host.docker.internal:19090:127.0.0.1 \
  -X POST http://host.docker.internal:19090/mcp/tools/call \
  -H 'Content-Type: application/json' \
  -d '{"tool":"echo","input":"called after NANDA discovery"}'
```

The same search and call pattern works for the built-in demo agents:

```sh
# Finds the registry that advertises the built-in reverse tool.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  'http://nanda-a.local:18080/search?tool=reverse'

# Calls the reverse tool through its host-published port.
curl --resolve enterprise-a.reverse.mcp.local:18087:127.0.0.1 \
  -X POST http://enterprise-a.reverse.mcp.local:18087/mcp/tools/call \
  -H 'Content-Type: application/json' \
  -d '{"tool":"reverse","input":"another discovered tool"}'
```

## 6. Create A Dynamic Registry

The built-in registries are started by compose, but the same binary can also create a new runtime registry without a Go edit. The dynamic registry:

- writes its runtime registry spec into `./artifacts`
- signs short-lived registry address and catalog VCs
- rotates those credentials every 15 seconds with 45 second TTLs
- joins NANDA index A through `POST /join`
- accepts runtime agent registrations through `POST /agents/register`

Start a second host Python agent for the dynamic registry in another terminal:

```sh
AGENT_ID=lab-python-agent TOOLS=title,length PORT=19091 uv run --python 3.12 /tmp/level8_python_agent.py
```

Launch the new registry:

```sh
# Starts a new runtime registry container and joins it to NANDA index A.
docker compose run -d --no-deps \
  --name level8-lab-registry \
  -p 18101:8080 \
  --entrypoint /shared-bin/enterprise-registry \
  enterprise-a-registry \
  --artifacts /artifacts \
  --logs /logs \
  --enterprise lab \
  --registry-name lab.registry.nanda.local \
  --registry-id lab-registry \
  --catalog-url http://host.docker.internal:18101/catalog \
  --facts-mode public \
  --description "Runtime lab registry for ad-hoc agent registration." \
  --join-index http://nanda-index-a:8080 \
  --addr :8080
```

Confirm the registry is live:

```sh
# Proves the dynamic registry process is healthy.
curl --resolve lab.registry.nanda.local:18101:127.0.0.1 \
  http://lab.registry.nanda.local:18101/healthz

# Shows the current signed catalog.
curl --resolve lab.registry.nanda.local:18101:127.0.0.1 \
  http://lab.registry.nanda.local:18101/catalog
```

Register the agent with the new registry:

```sh
# Registers the host Python agent with the dynamic registry.
curl --resolve lab.registry.nanda.local:18101:127.0.0.1 \
  -X POST http://lab.registry.nanda.local:18101/agents/register \
  -H 'Content-Type: application/json' \
  -d '{
    "id": "lab-python-agent",
    "name": "lab.python.agent.mcp.local",
    "endpoint": "http://host.docker.internal:19091",
    "tools": ["title", "length"]
  }'

# Shows the registry's registered agents.
curl --resolve lab.registry.nanda.local:18101:127.0.0.1 \
  http://lab.registry.nanda.local:18101/agents
```

Wait two seconds for the dynamic registry to refresh its NANDA join.

Find and call the dynamic registry's tool:

```sh
# Proves NANDA index A can find the dynamic registry by tool.
# The response includes the dynamically registered lab-python-agent.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  'http://nanda-a.local:18080/search?tool=title'

# Proves NANDA index A can find the dynamic registry by agent id.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  'http://nanda-a.local:18080/search?agent=lab-python-agent'

# Resolves the signed registry address credential from NANDA.
curl --resolve nanda-a.local:18080:127.0.0.1 \
  http://nanda-a.local:18080/resolve/lab.registry.nanda.local

# Fetches the signed AgentFacts-like catalog from the dynamic registry.
curl --resolve lab.registry.nanda.local:18101:127.0.0.1 \
  http://lab.registry.nanda.local:18101/catalog

# Calls the dynamically registered title tool.
curl --resolve host.docker.internal:19091:127.0.0.1 \
  -X POST http://host.docker.internal:19091/mcp/tools/call \
  -H 'Content-Type: application/json' \
  -d '{"tool":"title","input":"dynamic registry found this tool"}'

# Proves the dynamic agent rejects tools it did not advertise.
curl --resolve host.docker.internal:19091:127.0.0.1 \
  -X POST http://host.docker.internal:19091/mcp/tools/call \
  -H 'Content-Type: application/json' \
  -d '{"tool":"reverse","input":"should fail"}'
```

Watch the dynamic security signals:

```sh
# Shows the dynamic registry signing and rotating credentials.
tail -f ./logs/lab-registry.jsonl

# Shows NANDA accepting the dynamic registry join and refreshes.
tail -f ./logs/nanda-a.jsonl
```

Look for:

- `managed_registry_credentials_rotated`
- `registry_joined_quilt`
- `registry_refreshed_quilt`
- `agent_registered`
- `agent_registration_join_refreshed`
- `search_registries`

## 7. Full Automated Test

The automated test runs the base quilt plus e2e-only versions of the runtime registration and dynamic registry flows above. It uses the Go MCP agent binary as a stand-in, so the test does not require `uv` or a host Python process.

During the run, look for these progress lines:

- `ok: base quilt, federation, VC verification, revocation, CRDT, key rotation, and privacy checks passed`
- `ok: Enterprise A dynamically registered agent was found through NANDA and called`
- `ok: dynamic registry was discovered through NANDA and its registered tool was called`

```sh
# Runs the full Level 8 e2e and tears down compose at the end.
./scripts/test-e2e.sh
```

## Host Ports

- `18080`: NANDA index A
- `18086`: NANDA index B
- `18081`: Enterprise A registry
- `18082`: Enterprise B registry
- `18083`: private facts gateway
- `18084`: revocation authority
- `18085`: CRDT update bus
- `18087`: Enterprise A reverse MCP agent
- `18101`: optional dynamic lab registry
- `19090`: optional host Python agent for Enterprise A
- `19091`: optional host Python agent for dynamic registry
